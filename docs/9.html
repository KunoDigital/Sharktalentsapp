CRÍTICO #5 Presupuesto de costos por puesto = 20% del precio cobrado
Fase 3 — DECISIÓN TUYA
Es como darle un sobre con dinero a cada puesto para gastar (publicidad, IA, mensajes). El sobre tiene un techo: el 20% de lo que el cliente paga. Cuando se acaba, no se gasta más sin tu OK.
Síntoma (sin fix)Si un bug pone una llamada de IA en loop de reintentos (60/h × 24h × 30 días × $0.05 ≈ $2.160 USD silenciosos), o si la pauta de LinkedIn se va de mambo, los costos se comen el margen sin alarma.
Regla de negocio Costos totales del puesto ≤ 20% del precio cobrado al cliente.
Ejemplo: cobrás $1.500 → presupuesto total = $300 para repartir entre Anthropic + ZeptoMail + Twilio + pauta LinkedIn + storage.
CausaEl código mide y registra costos DESPUÉS de cada llamada (en JobCostEvents). Antes de llamar no consulta el acumulado del puesto vs el presupuesto. Y además: el precio cobrado no está modelado en la base (la tabla Jobs no tiene campo fee_usd).
Archivos • Tabla Jobs (agregar columna fee_usd)
• Frontend JobForm (input para cargar el precio al crear puesto)
• Frontend JobDetail (input "agregar gasto LinkedIn ads")
• functions/api/src/lib/costTracking.ts (agregar tipo "ads")
• functions/api/src/lib/budgetWatch.ts (NUEVO — calcula % consumido + dispara alertas)
• Cron/trigger que evalúa cada N min si algún puesto pasó el umbral
Fix 1. Agregar campo fee_usd en Jobs.
2. Vos cargás el precio cuando creás el puesto + gastos de LinkedIn cuando hagas pauta.
3. Después de cada gasto registrado en JobCostEvents: sumar acumulado del puesto, comparar contra fee_usd × 0.20.
4. Alertas escalonadas:
  • 80% → alerta tipo "warning" (email a vos).
  • 100% → alerta "critical" + genera post-mortem automático en el JobDetail: "se gastaron $X (Y% del fee) — desglose: Anthropic $A, email $E, WhatsApp $W, ads $L. Lo más alto: [tipo]. Sugerencias: [análisis IA opcional]".
5. NUNCA bloquea ningún flujo. El sistema avisa, registra, aprende.
Qué más toca • Dashboard del JobDetail muestra barra "gastado X de presupuesto Y" con color (verde <60%, naranja 60-80%, rojo >100%).
• Cuando se pasa: aparece tarjeta "Post-mortem del puesto" con el desglose y por qué.
• Ningún call site existente cambia. Solo agregamos un observador.
Decisiones cerradas ✅
1. ¿De dónde sale el precio? ✅ Agregar campo fee_usd en Jobs. El fee NO sale del CRM del cliente — sale del Trato (Deal) de Zoho cuando Cris lo crea. Por ahora vos lo cargás manual al crear el puesto en SharkTalents.

2. ¿LinkedIn entra en el 20%? ✅ Sí, carga manual (vos pegás el gasto en el JobDetail). Histórico: máximo $80 hasta ahora. Esto sirve para calibrar las alertas — un gasto de >$100 en LinkedIn ya debería ser raro.

3. ¿Qué pasa al 80% y al 100%? ✅ SOLO ALERTAS. NUNCA bloquear el flujo. Filosofía de Cris: "el flujo no debe detenerse, solo aprender qué hicimos mal o por qué el puesto es más costoso". Cuando se pasa del 100%, el sistema genera un post-mortem automático identificando qué tipos de costo se dispararon (ej: "se generó IA 12 veces para tech questions cuando lo normal es 1-2 = bug"). Eso queda en el dashboard del puesto para aprender.

4. Puestos sin fee_usd cargado. Filosofía coherente con (3): seguir trackeando, no bloquear, no alertar hasta que se cargue. Cuando se carga, el sistema retroactivamente calcula si ya se pasó del 20% y dispara alertas si corresponde.
Validación ANTES • Mirar últimas 30 entradas en JobCostEvents. Sumar por puesto. ¿Cuánto es el peor caso real?
• Comparar con un fee típico tuyo para ver qué porcentaje da hoy.
Validación DESPUÉS 1) Crear puesto con fee_usd=100. Umbral = $20. Disparar generaciones hasta cruzar $16 (80%) → debe llegar alerta warning. Cruzar $20 (100%) → alerta critical + post-mortem aparece en JobDetail.
2) Verificar que el flujo SIGUE funcionando después del 100% (no bloquea).
3) Crear puesto SIN fee_usd. Gastar. No debe alertar. Cargar el fee retroactivo → si ya pasó el 80%, debe alertar al momento de cargarlo.
Riesgo de romperBAJO Como no bloquea nada, no rompe ningún flujo. Solo agrega observabilidad.
Cómo deshago si rompeEnv var BUDGET_ALERTS_ENABLED=false apaga las alertas en caliente sin redeploy. El tracking en JobCostEvents sigue normal.
Estimación ~1.5h código (campo fee + budgetWatch + alertas + dashboard barra) + 30 min post-mortem auto + 30 min prueba ≈ 2.5 horas

🆕 Fase 3.5 — Webhook Recruit → SharkTalents al registrarse candidato

NUEVO Pedido de Cristian — Recruit avisa a SharkTalents cuando entra candidato nuevo
Fase 3.5
Hoy Recruit es como una recepcionista que recibe a candidatos pero no avisa a las áreas correspondientes. Lo que pedís: cuando entra un candidato, que la recepcionista llame a SharkTalents y diga "che, llegó Juan para el puesto X, manejalo vos".
PedidoCuando un candidato se registra en Zoho Recruit, Recruit dispara un webhook a SharkTalents. SharkTalents crea automáticamente la Application en el Job correspondiente. A partir de ahí, todos los mensajes (email, WhatsApp) los manda SharkTalents, no Recruit. Cierra la arquitectura "Recruit = puerta de entrada + LinkedIn gratis, SharkTalents = todo el resto".
Estado actual ✅ Webhook EXISTE en functions/api/src/features/zohoRecruitWebhook.ts.
✅ Escucha 3 eventos: candidate.status_changed, candidate.hired, candidate.rejected.
❌ NO escucha el evento "candidato se registró por primera vez".
❌ NO hay workflow rule configurada en Zoho Recruit Console.
Fix Parte backend (Claude):
1. Agregar evento nuevo candidate.created (o application.created) al webhook handler.
2. El handler busca el Job en SharkTalents por recruit_job_id, encuentra el candidato por recruit_candidate_id (o lo crea si no existe), e inserta una Application nueva en estado inicial.
3. Disparar evento application.created al outbox para que SharkTalents mande el primer email/WhatsApp al candidato.

Parte manual (Cristian / vos):
4. En Zoho Recruit Console → Setup → Automation → Workflow Rules → Create New.
5. Trigger: "Candidate is created" o "Candidate is associated to Job Opening".
6. Action: "Custom Function" o "Webhook" → POST a https://app.sharktalents.ai/server/api/api/webhooks/zoho-recruit con body {event_id, event_type:"candidate.created", candidate_id, recruit_job_id} y header X-Zoho-Recruit-Secret.
Qué más toca • Frontend NO cambia.
• Si el evento se ejecuta para candidato que YA existe (por aplicación previa al mismo job), debe ser idempotente — no duplica Application.
• El nuevo evento se integra con el sistema de outbox (mismo patrón usado por candidate.status_changed).
• Importante: el bug #15 de auditoría (idempotencia del webhook check-then-act) afecta también a este nuevo evento. Mejor arreglar #15 ANTES o EN PARALELO con esta feature.
Validación ANTES 1) Confirmar que Zoho Recruit soporta webhook al evento "candidate created". Algunos planes de Recruit limitan workflow rules.
2) Confirmar que el campo recruit_job_id en SharkTalents Jobs está siempre seteado para puestos publicados en Recruit (sí: ya está, según memoria recruit-sync).
3) Crear un candidato de prueba manual en Recruit + ver si se genera Application en SharkTalents (debería NO generarse → confirma estado actual).
Validación DESPUÉS 1) Después de configurar el workflow rule + deploy del backend, crear candidato nuevo manualmente en Recruit.
2) En menos de 30 seg debe aparecer la Application en SharkTalents.
3) El primer email/WhatsApp llega al candidato desde SharkTalents (no desde Recruit).
4) Test de idempotencia: disparar 2 veces el mismo webhook → solo una Application.
Riesgo de romper MEDIO
• Si el workflow rule de Recruit dispara más eventos de los esperados (ej. también para cambios de campos), se generan Applications duplicadas. Mitigación: idempotency por event_id.
• Si el campo recruit_job_id no está en algún Job antiguo, el handler no encuentra el Job y la Application no se crea. Mitigación: alerta clara al admin (en SystemAlerts) cuando esto pase.
• Solapamiento con bug #15 (idempotencia rota): si no se arregla, puede haber duplicados en escenarios de retry.
Cómo deshago si rompe • En Zoho Recruit Console: desactivar el workflow rule (no requiere redeploy).
• En SharkTalents: revertir el handler nuevo. Vuelve al estado actual (solo 3 eventos).
Decisión tuya
1. ¿Cuándo hacemos esto?
• Antes del go-live 2026-05-15 (eso ya pasó — confusión de fecha) → ALTA prioridad ahora.
• Después de Fase 1-2 de auditoría (recomendado: arreglar #15 primero porque comparten código).
